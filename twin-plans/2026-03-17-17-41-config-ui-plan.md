## Análise Técnica

### Estado Atual
O agente de impressão (`print-agent/`) é uma aplicação Node.js CLI que:
- Carrega configurações via `.env` (API_URL, API_TOKEN, PRINTER_IDENTIFIER, etc.)
- Roda em loop de polling sem interface gráfica
- Requer edição manual do `.env` para configurar

### Problema
Configurar o agente requer:
1. Editar `.env` manualmente
2. Conhecimento técnico (URLs, tokens, identificadores UUID)
3. Reiniciar o agente após mudanças
4. Sem validação visual das configurações

### Solução Arquitetural
**Web UI local com Express.js** servindo interface de configuração

**Por quê?**
- ✅ Express já é comum no ecossistema Node.js
- ✅ Interface web acessível via navegador (localhost:3001)
- ✅ Não interfere com o polling principal
- ✅ Permite validação antes de salvar
- ✅ Usuário pode registrar impressora sem usar curl

**Restrições:**
- Deve ser **local-only** (127.0.0.1, não expor na rede)
- Mínimo de dependências
- Não pode interferir com o polling existente

### Riscos Técnicos e Mitigações

**1. Porta 3001 já em uso**
- **Risco**: Conflito de porta
- **Mitigação**: Detectar porta em uso e tentar 3002, 3003, etc. (loop até encontrar livre)

**2. Segurança do API_TOKEN**
- **Risco**: Token exposto na UI
- **Mitigação**: 
  - Servidor bind apenas em 127.0.0.1 (local-only)
  - Mostrar apenas últimos 4 caracteres no GET /api/config
  - Avisar usuário para não compartilhar tela

**3. Permissões de escrita no .env**
- **Risco**: Falha ao salvar .env (Windows pode bloquear)
- **Mitigação**: Try-catch com mensagem de erro clara, verificar permissões

**4. Navegador não abre automaticamente**
- **Risco**: Usuário não sabe acessar
- **Mitigação**: 
  - Mostrar URL no console em destaque
  - Instruções claras: "Abra http://localhost:3001 no navegador"

**5. Validação de campos**
- **Risco**: Configurações inválidas salvam e quebram agente
- **Mitigação**: 
  - Validação no frontend (URLs, UUIDs)
  - Validação no backend antes de salvar
  - Testar conexão obrigatório antes de salvar
